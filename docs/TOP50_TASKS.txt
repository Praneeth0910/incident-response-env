================================================================
TOP 50 REAL-WORLD INCIDENT RESPONSE TASKS
incident-response-env · Task Design Reference
Based on actual incidents at Netflix, Uber, Cloudflare, AWS,
Meta, Google, Stripe, GitHub, Shopify, and Airbnb.
================================================================

Each entry includes:
  ID         : task identifier slug
  DIFFICULTY : easy / medium / hard
  FAULT SVC  : which service is actually broken
  FAULT TYPE : the underlying failure mode
  RED HERRING: which service looks suspicious but isn't
  ALERT      : what the on-call engineer sees
  DESCRIPTION: full scenario with real-world context
  IDEAL STEPS: expected optimal step count
  FIX        : correct remediation action

EASY TASKS (single service, no red herrings, clear signal)

[01]
ID          : task_cpu_spike_auth
DIFFICULTY  : easy
FAULT SVC   : auth-service
FAULT TYPE  : cpu_spike
RED HERRING : none
ALERT       : ALERT: Login latency p99 > 8s. Auth service CPU at 99%. Users cannot sign in.
DESCRIPTION : A hot loop was introduced in the JWT validation code during a minor
              refactor. Auth-service is pegging CPU at 99%, causing all login
              requests to time out. The fix is a restart to clear the runaway
              thread, followed by a hotfix deployment. This mirrors the 2021
              Fastly outage pattern where a single bad config triggered global
              service degradation through a CPU-bound loop.
IDEAL STEPS : 3
FIX         : restart_service → auth-service, then declare_rca

[02]
ID          : task_disk_full_postgres
DIFFICULTY  : easy
FAULT SVC   : postgres-db
FAULT TYPE  : disk_full
RED HERRING : none
ALERT       : ALERT: Write operations failing. postgres-db throwing ENOSPC errors. Order creation down 100%.
DESCRIPTION : The database transaction log (WAL) grew unbounded after auto-vacuum

IDEAL STEPS : 4
FIX         : run_db_query → postgres-db to confirm, declare_rca → postgres-db

HARD TASKS (complex cascade, 2+ red herrings, requires DB query, 8-12 ideal steps)
----------------------------------------------------------------

[19]
ID          : task_multi_fault_hard
DIFFICULTY  : hard
FAULT SVC   : postgres-db  (primary fault: slow queries)
FAULT TYPE  : dual_fault    (+ redis eviction causing cache thrash)
RED HERRING : order-service (CPU spike from retry storms)
ALERT       : ALERT: Full system degradation. p99 latency 12s. 4 services alarming. On-call paged.
DESCRIPTION : Two simultaneous faults: (1) A missing index on the sessions table
              causes postgres-db to full-scan 80M rows per auth request. (2)
              Redis-cache hit its memory limit and switched to aggressive
              eviction, causing cache thrash — cached items evicted faster
              than they can be repopulated. Order-service enters a retry
              storm trying to compensate for both, pegging CPU at 94%.
              The agent must identify BOTH root causes to declare correct RCA.
              This dual-fault pattern mirrors the AWS us-east-1 degradation
              of November 2020 where multiple systems failed simultaneously.
IDEAL STEPS : 10
FIX         : run_db_query → postgres-db (confirm), rollback_deployment → redis-cache, declare_rca → postgres-db
